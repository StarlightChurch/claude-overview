핵심 문제 · The Consistency Problem

왜 같은 질문에 매번 다른 답이 나오나?

커스텀 Instructions를 배우기 전에, 왜 이것이 필요한지 직접 눈으로 확인합니다.

일관성 없는 답변이 주는 실제 피해

❌ Instructions 없이 — 매번 다른 답변
월요일: "이 이메일 다시 써줘"
→ 매우 격식체, 3단락, 영문 혼용
수요일: "이 이메일 다시 써줘"
→ 친근한 어체, 1단락, 이모지 포함
금요일: "이 이메일 다시 써줘"
→ 중간 어체, 2단락, 요청 없이 PS 추가
→ 3번 모두 다른 결과. 어느 게 맞는지 모름.
✅ Instructions 있는 경우 — 항상 일관된 답변
월요일: "이 이메일 다시 써줘"
→ 비즈니스 캐주얼, 200자, 제목+본문 형식
수요일: "이 이메일 다시 써줘"
→ 비즈니스 캐주얼, 200자, 제목+본문 형식
금요일: "이 이메일 다시 써줘"
→ 비즈니스 캐주얼, 200자, 제목+본문 형식
→ 항상 동일한 기준. 팀 전체가 같은 품질.

일관성이 특히 중요한 3가지 상황

🏢
브랜드 커뮤니케이션
고객에게 보내는 이메일·공지·SNS가 담당자마다 다른 어조여선 안 됩니다.
📊
반복 업무 자동화
주간 보고서, 회의록 요약, 데이터 분석이 매주 같은 형식이어야 비교·추적이 가능합니다.
👥
팀 공유 프로젝트
여러 팀원이 같은 프로젝트를 쓸 때, 누가 질문해도 같은 기준의 답이 나와야 합니다.
🎓
교육·학습
학생 수준에 맞게 항상 같은 방식으로 설명해야 혼란이 없습니다.
⚖️
규정 준수
법적·컴플라이언스 기준이 있는 산업에서는 모든 답변이 규정 내에 있어야 합니다.
✍️
창작·글쓰기
연재물이나 브랜드 콘텐츠는 시리즈 전체에서 동일한 문체가 유지되어야 합니다.

Instructions가 작동하는 3개 레이어

레이어 1 — 가장 강력
📁 Project Instructions
이 프로젝트의 모든 대화에 항상 적용. 역할·규칙·금지사항의 핵심. 오늘 배울 주제.
레이어 2
👤 Profile Preferences (Day 3에서 배움)
계정 전체에 적용. 개인 스타일·언어·기본 행동 방침. Project Instructions이 우선.
레이어 3 — 가장 약함
💬 대화 내 개별 프롬프트
그 대화에서만 일시적 적용. Instructions와 충돌 시 Instructions 우선.
🔑 핵심 원칙: Instructions는 "계약서"다
클로드는 Instructions를 마치 고용 계약서처럼 대합니다. "항상 한국어로", "300자 이내로", "확실하지 않으면 물어봐라" — 이런 조항들은 명시적으로 번복하지 않는 한 지속 적용됩니다. 구체적이고 명확한 규칙일수록 더 강력하게 작동합니다.
✅ 적용방법 — 현재 일관성 문제 진단
오늘 심화 학습을 시작하기 전에, 지금 내 프로젝트의 일관성 문제를 먼저 파악합니다.

# 현재 Instructions의 문제점을 파악하고 개선 방향을 잡는 진단 프롬프트 아래 같은 요청을 3번 연속으로 해서 일관성을 테스트해줘. 매번 같은 답이 나오는지 확인하고 싶어. [테스트 요청 — 3번 연속으로 아래를 그대로 실행해줘] 첫 번째: "[내 업무에서 자주 하는 요청을 입력]" 두 번째: [위와 완전히 같은 문장을 다시 입력] 세 번째: [위와 완전히 같은 문장을 다시 입력] 테스트 후 스스로 평가해줘: ① 3번의 답변에서 차이가 있다면 무엇인가? ② 차이가 생긴 원인은 무엇인가? ③ Instructions에 어떤 규칙을 추가하면 이 차이를 없앨 수 있는가? ④ 일관성을 높이기 위한 추천 조치 3가지
Section 1 · Instructions 해부학

📋 Instructions 완전 해부 — 7가지 구성 블록

효과적인 Instructions는 7가지 구성 요소를 갖고 있습니다. 각각의 역할과 작성 방법을 배웁니다.

완성형 Instructions의 7가지 블록
1
역할 정의 (Role)
클로드의 정체성·전문성·관점을 설정. "너는 ___ 이다."
2
목적·맥락 (Purpose)
이 프로젝트가 왜 존재하는지, 어떤 상황에서 사용되는지.
3
톤 & 어조 (Tone)
언어, 말투, 격식 수준, 감정적 온도를 구체적으로 지정.
4
행동 규칙 (Rules)
해야 할 것과 절대 하지 말아야 할 것의 명확한 목록.
5
출력 형식 (Format)
결과물의 구조, 길이, 섹션 순서를 고정.
6
지식 기반 안내 (KB)
업로드된 문서의 역할과 참조 우선순위 설명.
7
예외 처리 (Edge Cases)
모르는 것, 범위 밖 요청, 충돌 상황에서 어떻게 행동할지.

7블록을 모두 갖춘 마스터 Instructions 구조

📋 마스터 Instructions 뼈대 — 복사 후 내용 채우기
## 역할 (Role) 너는 [구체적 역할 + 경력/전문성]이다. [이 역할의 핵심 관점이나 철학]. ## 목적 (Purpose) 이 프로젝트의 목적은 [목적]. 주요 사용 상황: [상황 1], [상황 2], [상황 3]. ## 톤 & 어조 (Tone) - 언어: 항상 한국어 - 격식 수준: [격식체/비즈니스 캐주얼/친근한 어체] - 특별 어조 지침: [예: 전문 용어 풀어서 설명, 이모지 사용 금지] ## 행동 규칙 (Rules) ✅ 반드시 해야 할 것: - [규칙 1] - [규칙 2] ❌ 절대 하지 말 것: - [금지 1] - [금지 2] ## 출력 형식 (Format) [작업 유형]의 경우: - 구조: [섹션 1] → [섹션 2] → [섹션 3] - 길이: [최대 X자 또는 X줄] - 형식: [bullet / 번호 목록 / 산문 등] ## 지식 기반 (Knowledge Base) [파일명 1]: [이 파일이 무엇이고 언제 참조할지] [파일명 2]: [이 파일이 무엇이고 언제 참조할지] 문서에 없는 내용은 "(미확인)"으로 표시. ## 예외 처리 (Edge Cases) - 범위 밖 요청: [어떻게 처리할지] - 불확실한 경우: [어떻게 처리할지] - 충돌하는 정보: [어떻게 처리할지]

얼마나 길어야 하는가? — 최적 길이 가이드

길이적합한 경우주의사항
50~100자 (짧음)단순 반복 작업, 1가지 규칙만 필요한 경우빠르게 작성, 확실히 따름
200~400자 (보통)일반적인 업무 프로젝트, 5~7개 규칙대부분의 상황에 최적
500~800자 (길이)복잡한 규정·정책, 팀 공유 프로젝트작성 신중히, 테스트 필수
800자 초과특수 산업(의료·법률·금융 컴플라이언스)일관성 저하 위험, 축약 권장
⚠️ Instructions 작성의 4가지 함정
함정 1 — 모순 규칙: "간결하게" + "상세하게" → 어느 것을 따를지 모름. 우선순위를 명시하거나 상황별로 분리하세요.
함정 2 — 추상적 지시: "잘 써줘", "전문적으로" → 기준이 없어서 일관성이 없음. "최대 300자", "기술 용어 사용 금지" 처럼 구체적으로.
함정 3 — 과도한 복잡성: 규칙이 너무 많으면 충돌 발생. 핵심 5~7개만 엄선하세요.
함정 4 — 테스트 생략: 만들고 바로 사용하면 숨은 문제를 모름. 반드시 같은 질문 3번으로 일관성 테스트.

# 내 답변을 바탕으로 7블록 구조의 Instructions 생성 내 프로젝트를 위한 완성형 Instructions를 7블록 구조로 작성해줘. [블록 1 — 역할] [예: 10년 경력 기술 블로그 에디터, B2B SaaS 분야 전문] [블록 2 — 목적] [예: 개발팀 주간 기술 블로그 초안 작성, 독자는 비개발자 임원] [블록 3 — 톤] [예: 전문적이지만 딱딱하지 않게, 기술 용어는 괄호 안에 풀어서] [블록 4 — 규칙] [예: 반드시 실제 예시 포함, AI 티나는 표현 금지] [블록 5 — 형식] [예: 제목+요약+본문(H2 3개)+결론, 총 800~1200자] [블록 6 — KB] [예: style_guide.pdf=우리 브랜드 가이드라인, past_posts.pdf=이전 글 모음] [블록 7 — 예외] [예: 기술 사실 불확실 시 "(검토 필요)" 표시] 위 7가지를 반영한 완성형 Instructions를 작성해줘. 바로 프로젝트 Instructions 창에 붙여넣을 수 있게 깔끔하게.
Section 2 · 톤 & 어조 지침

🎭 톤 & 어조 — 클로드의 말투를 고정하기

같은 내용이라도 말투가 다르면 완전히 다른 결과물이 됩니다. 톤 지침을 구체적으로 설정하는 방법을 배웁니다.

톤 지침의 4가지 차원

차원 1: 격식 수준 (Formality)
격식체: "제출해 주시기 바랍니다" — 공문서, 법률, 의료
비즈니스 캐주얼: "제출해 주세요" — 일반 업무, B2B
친근한 어체: "보내줘요", "이렇게 해봐요" — SNS, 커뮤니티
→ 반드시 구체적 예시 문장으로 지정하세요. "전문적으로"는 기준이 없습니다.
차원 2: 감정 온도 (Emotional Warmth)
차갑고 중립적: 데이터·통계·기술 문서에 적합
따뜻하고 공감적: 고객 응대, HR, 교육에 적합
활기차고 열정적: 마케팅, 이벤트, 소셜 콘텐츠에 적합
→ "공감하는 어조로" 보다 "고객의 불편을 먼저 인정하고 해결책을 제시"가 더 정확합니다.
차원 3: 언어 규칙 (Language Rules)
영어 혼용 금지 / 필수 사용
전문 용어 사용 허용 / 풀어서 설명 필수
이모지 사용 / 금지
단어 선택 규칙: "구매" 대신 "결제", "고객님" 대신 "회원님" 등
차원 4: 금지 표현 목록 (Banned Phrases)
클로드가 기본적으로 자주 쓰는 표현 중 브랜드에 맞지 않는 것을 명시적으로 금지하세요.
예: "물론입니다", "좋은 질문이에요", "확실히", "기억하세요", "중요한 것은"

좋은 톤 지침 vs 나쁜 톤 지침

❌ 나쁜 톤 지침 — 모호하고 주관적
"전문적으로 써줘" — 기준 없음
"친절하게 답해줘" — 클로드의 해석에 따라 달라짐
"너무 딱딱하지 않게" — 정도를 알 수 없음
"자연스러운 어투로" — 기준이 없음
✅ 좋은 톤 지침 — 구체적이고 측정 가능
"경어체 사용, '해주세요' 어미"
"고객 불만 시 1문장 사과 → 해결책 제시"
"영어 전문 용어는 항상 (한글 병기)"
"이모지 금지. '물론'·'확실히' 금지"

Section 3 · 행동 규칙 지침

⚙️ 행동 규칙 — 해야 할 것과 하지 말아야 할 것

행동 규칙은 Instructions에서 가장 강력한 부분입니다. "해야 할 것"과 "하지 말아야 할 것"을 모두 명시해야 완전합니다.

효과적인 행동 규칙의 3가지 유형

유형설명예시
✅ 필수 행동
항상 해야 할 것		 클로드가 기본적으로 안 하지만 내가 원하는 행동 "매 답변 마지막에 다음 단계 제안 포함", "수치 인용 시 출처 명시"
❌ 금지 행동
절대 하지 말 것		 클로드가 자주 하지만 내가 원하지 않는 기본 행동 "'물론'으로 시작 금지", "확인 없이 개인 추천 금지"
🔀 조건부 행동
상황에 따라		 특정 조건에서만 적용되는 규칙 "요청이 범위 밖일 경우 → 이유 설명 후 범위 내 대안 제시"

클로드의 기본 습관 — 고쳐야 할 것 vs 살려야 할 것

🔧 자주 고쳐야 하는 기본 습관
"좋은 질문이에요!"로 칭찬하며 시작
답변 끝에 길고 불필요한 요약 추가
모든 것을 bullet point로 나열
확인 없이 모든 요청에 과도하게 응함
불확실한 내용도 자신 있게 말함
"물론", "확실히", "중요한 것은" 남발
💎 살려야 하는 장점
논리적 구조로 설명하는 능력
다양한 관점을 제시하는 균형감
애매한 요청 시 명확히 물어보는 것
복잡한 내용을 단계별로 풀어내는 것
한계를 솔직하게 인정하는 것

# 내 프로젝트의 행동 규칙을 최적화하는 Instructions 생성 내 프로젝트의 행동 규칙 섹션을 만들어줘. [내가 원하는 것 — 항상 해주길 바라는 것들] ① [예: 카피는 항상 3가지 버전 제시 (보수/중간/과감)] ② [예: 타겟 독자의 감정을 먼저 언급하고 시작] ③ [예: 경쟁사 분석 시 데이터 근거 포함] [내가 원하지 않는 것 — 하지 말았으면 하는 것들] ① [예: 교과서 같은 정의나 배경 설명로 시작 금지] ② [예: "이것이 중요합니다"처럼 교훈적 마무리 금지] ③ [예: 모호한 형용사("훌륭한", "최고의") 사용 금지] [상황별 특별 규칙] [예: 경쟁사 언급 요청 시 → 직접 비교 대신 우리 강점 부각] 이를 바탕으로: ① 바로 Instructions에 넣을 행동 규칙 섹션 ② 각 규칙이 실제로 효과적으로 작동하는지 테스트할 예시 질문 3개

# 클로드가 모르는 것을 어떻게 처리할지 명확히 지정 이 프로젝트에서 클로드가 불확실한 상황을 처리하는 방법을 지정해줘. [불확실성 유형별 처리 규칙을 만들어줘] 유형 1 — 업로드 문서에 없는 내용을 묻는 경우 내가 원하는 처리: [예: "(문서 미확인)" 표시 + 클로드 일반 지식으로 보충 답변 + "확인 필요" 안내] 유형 2 — 정답이 여러 개인 판단 요청 내가 원하는 처리: [예: 가장 일반적 방법 1가지 추천 + 대안 2가지 간략 제시] 유형 3 — 범위 밖 요청 (이 프로젝트 목적과 무관한 요청) 내가 원하는 처리: [예: 범위 밖임을 정중히 설명 + 해당 요청에 적합한 방법 제안] 유형 4 — 충돌하는 정보 (문서 간 내용이 다를 때) 내가 원하는 처리: [예: 두 버전 모두 제시 + 어느 것이 최신인지 확인 요청] 위 4가지를 포함한 예외 처리 Instructions 섹션을 완성해줘.
Section 4 · 출력 형식 지침

📐 출력 형식 — 결과물의 구조를 고정하기

결과물의 모양을 미리 정해두면 매번 수정하는 시간이 사라집니다. 형식 지침 작성의 모든 것을 배웁니다.

형식 지침의 4가지 차원

차원 1: 구조 (Structure)
결과물이 어떤 섹션으로 구성되어야 하는가?
예: 이메일 = [제목 3개 → 본문 → CTA 1개]
보고서 = [요약 → 배경 → 분석 → 결론 → 다음 단계]
차원 2: 길이 (Length)
각 섹션 또는 전체의 길이를 구체적으로 지정
좋음: "전체 300자 이내", "본문 2단락, 각 3문장"
나쁨: "간결하게", "적당한 길이로"
차원 3: 형식 요소 (Elements)
사용할 형식 요소와 그 규칙
불릿 포인트 / 번호 목록 / 표 / 코드 블록 / 이모지
언제 쓰는지, 얼마나 쓰는지도 명시
차원 4: 시작·끝 패턴 (Opening & Closing)
답변이 어떻게 시작하고 끝나야 하는가?
금지: "물론이죠!으로 시작 금지", "요약으로 마무리 금지"
필수: "핵심 결론을 첫 문장에", "다음 단계 제안으로 마무리"

Section 6 · 디버깅 & 고급 전략

🔬 Instructions 디버깅 & 고급 전략

만든 Instructions가 기대대로 작동하지 않을 때 원인을 찾고 고치는 방법, 그리고 Instructions를 지속적으로 발전시키는 전략을 배웁니다.

Instructions 디버깅 5단계

진단: 어떤 규칙이 지켜지지 않는가?
문제가 되는 특정 규칙을 격리하세요. "전체 Instructions가 문제"가 아니라 "이 규칙이 작동 안 함"을 특정해야 수정이 가능합니다.
원인 분류: 모호 vs 충돌 vs 위치 문제
모호: "간결하게"처럼 기준이 없는 규칙 → 수치로 구체화
충돌: 두 규칙이 서로 반대 → 우선순위 명시
위치: Instructions 너무 뒤에 있어 묻힘 → 핵심 규칙을 앞으로
단위 테스트: 규칙 하나씩 분리 테스트
문제 있는 규칙만 별도로 꺼내서 간단한 Instructions로 테스트합니다. 단독으로는 작동하는데 전체에서 안 된다면 충돌 문제입니다.
강화: 규칙 앞에 강조 표현 추가
중요한 규칙 앞에 "반드시", "절대", "항상", "예외 없이"를 추가하면 클로드가 더 강하게 따릅니다. 또는 규칙을 Instructions 맨 앞으로 이동하세요.
검증: 수정 후 일관성 3회 테스트
수정 후 반드시 같은 요청을 3번 해서 일관성을 확인합니다. 2번 잘 되고 1번 실패하면 아직 불완전한 것입니다.
🎭 실습 시나리오 5
상황: Instructions에 "항상 3개 버전 제시"라고 했는데 가끔 1개만 줍니다. 특히 길이가 긴 요청에서 이 규칙이 무시되는 것 같습니다.
목표: 규칙 불이행 원인 파악 + 강화된 Instructions로 수정

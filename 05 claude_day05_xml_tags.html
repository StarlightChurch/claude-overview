Section 1 · 기초 개념

XML 태그란? — 왜 클로드에서 특별한가

XML 태그는 HTML 태그처럼 생겼지만, 클로드와 대화할 때 "정보의 종류를 구분하는 표지판" 역할을 합니다.

XML 태그 = 프롬프트의 "파일 분류 시스템"

클로드에게 긴 프롬프트를 보낼 때 가장 큰 문제는 정보가 뒤섞이는 것입니다. 배경 맥락인지, 지켜야 할 규칙인지, 분석할 문서인지 — 클로드가 헷갈리면 엉뚱한 답을 줍니다.

XML 태그는 각 정보에 "이건 맥락입니다", "이건 지시사항입니다", "이건 분석할 문서입니다"라는 라벨을 붙여줍니다. 마치 회사에서 서류마다 폴더 색깔로 구분하는 것과 같습니다.

XML 태그 기본 구조
<context>
  여기에 배경 정보를 씁니다.
  클로드가 이 영역의 내용을 "맥락"으로 인식합니다.
</context>

<instruction>
  여기에 지시사항을 씁니다.
  클로드가 이 영역의 내용을 "해야 할 일"로 인식합니다.
</instruction>

클로드가 XML 태그를 특별하게 처리하는 이유

Anthropic이 클로드를 학습시킬 때 XML 태그를 이용한 구조화된 데이터를 많이 사용했습니다. 그 결과 클로드는 다른 AI 모델보다 XML 태그를 훨씬 정확하게 파싱(인식·해석)합니다.

❌ 태그 없는 프롬프트
문제: 정보가 뒤섞여 클로드가 무엇이 배경이고 무엇이 지시인지 혼동

나는 마케터야. 경쟁사 분석이 필요해. 우리 회사는 B2B SaaS야. 간결하게 써줘. 아래 데이터를 봐줘. 지난 분기 매출 20% 하락. 요약해줘.
✅ 태그 있는 프롬프트
효과: 각 정보의 역할이 명확해서 클로드가 정확하게 처리

<context>나는 B2B SaaS 마케터</context>
<data>지난 분기 매출 20% 하락</data>
<instruction>경쟁사 분석 + 원인 추정</instruction>
<format>간결하게 3개 항목</format>

XML 태그가 만드는 3가지 효과

정보 혼동 방지 (Hallucination 감소)
참고 문서와 지시사항을 태그로 분리하면 클로드가 문서 내용을 지시사항으로 오해하거나, 지시사항을 인용하는 실수를 줄입니다. Anthropic 연구에 따르면 XML 구조화로 허위 정보 생성이 최대 40% 감소합니다.
복잡한 프롬프트도 일관된 품질 유지
여러 문서, 여러 지시사항, 여러 제약을 섞어도 태그로 구획하면 클로드가 각각을 독립적으로 처리합니다. 프롬프트가 길어져도 품질이 떨어지지 않습니다.
재사용 가능한 템플릿 제작 가능
태그 구조를 유지하면서 <input> 안의 내용만 바꾸면 같은 프롬프트 구조를 반복 사용할 수 있습니다. 업무 자동화의 첫 걸음입니다.
💡 언제 XML 태그가 필요한가?
필요할 때: 프롬프트에 여러 종류의 정보(맥락+지시+문서+예시)가 섞여 있을 때, 긴 문서를 첨부할 때, 동일 구조를 반복 사용할 때, 출력 형식을 정밀하게 제어할 때

필요 없을 때: 간단한 한 줄 질문, 빠른 아이디어 요청, 일상적 대화
✅ 적용방법 — XML 태그 첫 경험

Section 2 · 문법 기초

XML 태그 문법 — 작성 규칙과 구조

HTML을 써본 적 없어도 괜찮습니다. 규칙은 딱 4가지뿐입니다.

규칙 1: 여는 태그와 닫는 태그 짝 맞추기

# 올바른 형식
<태그이름> 내용 </태그이름>

# 여러 줄도 가능
<context>
  첫 번째 줄
  두 번째 줄
  세 번째 줄
</context>

# ❌ 잘못된 형식 — 닫는 태그 없음 (하지 마세요)
<context> 내용 ← 닫는 태그 </context> 없음 → 오류

규칙 2: 태그 이름은 영문·언더바만, 공백 금지

# ✅ 올바른 태그 이름
<context>      <user_data>     <document1>

# ❌ 잘못된 태그 이름
<user data>    ← 공백 포함 (오류)
<사용자맥락>   ← 한글 사용 (오류)

규칙 3: 태그 이름은 직관적으로 — 아무 이름이나 만들 수 있다

클로드는 특정 태그 이름에 고정된 의미가 없습니다. <context>든 <background>든 <myinfo>든 — 이름 자체보다 태그 이름이 설명적이고 일관성이 있는 것이 중요합니다.

💡 태그 이름 짓기 원칙
태그 이름만 보고 "이 안에 어떤 종류의 정보가 있겠다"를 알 수 있어야 합니다.
좋은 예: <context> <instruction> <document> <example> <output_format>
나쁜 예: <a> <stuff> <info1>

규칙 4: 속성(Attribute) — 태그에 메타 정보 추가하기

# 속성으로 태그에 추가 정보 붙이기
<document index="1" source="2024 Annual Report">
  문서 내용...
</document>

<document index="2" source="Q3 Sales Data">
  문서 내용...
</document>

# 여러 문서를 구분할 때 특히 유용합니다
⚠️ 자주 하는 실수 3가지
실수 1: 여는 태그만 쓰고 닫는 태그 빠뜨리기 → </태그이름> 반드시 쌍 맞추기
실수 2: 태그 이름에 공백 넣기 → <user data> ❌ → <user_data> ✅
실수 3: 하나의 프롬프트에 너무 많은 태그 → 3~5개가 최적. 단순한 내용에는 태그 불필요
✅ 적용방법 — 태그 문법 연습
아래 프롬프트를 클로드에게 그대로 입력하면 태그 구조를 올바르게 인식하는지 확인할 수 있습니다.
P-02복사해서 바로 쓰는 프롬프트 — 태그 문법 확인 테스트
# 이 프롬프트를 그대로 입력해서 태그 인식을 확인하세요 <task> 아래 각 태그 안의 내용이 어떤 종류의 정보인지 태그 이름을 기준으로 설명해줘. 그리고 내가 XML 태그를 올바르게 사용하고 있는지 평가해줘. </task> <background>나는 콘텐츠 마케터야. 인스타그램 팔로워를 늘리고 싶어.</background> <data>현재 팔로워: 500명 / 게시물 주기: 주 1회 / 주제: 음식</data> <goal>3개월 안에 팔로워 5,000명 달성</goal> <constraint>하루 30분 이하로 운영 가능해야 함</constraint>

Section 4 · 고급 구조

태그 중첩(Nesting) — 계층 구조 만들기

태그 안에 태그를 넣어 복잡한 정보를 체계적으로 정리합니다. 특히 여러 문서를 다룰 때 필수입니다.

중첩 태그의 기본 패턴

여러 문서 중첩 구조
<documents>
  <document index="1" source="2025 사업계획서">
    2025년 주요 목표: 매출 10억 달성, 팀 규모 20명...
  </document>

  <document index="2" source="Q1 실적 보고서">
    Q1 매출: 2억 3천만원, 팀원: 8명...
  </document>
</documents>

<instruction>
  document 1의 목표와 document 2의 실제 결과를 비교해서
  목표 달성률과 격차 원인을 분석해줘.
</instruction>
여러 예시 중첩 구조
<examples>
  <example type="good">
    <input>내일 회의 리마인드해줘</input>
    <output>📅 내일 오후 2시 주간 팀 회의 — Zoom 링크: ...</output>
  </example>
  <example type="bad">
    이렇게 쓰지 말 것: "안녕하세요! 내일 중요한 회의가 있음을 알려드립니다..."
  </example>
</examples>

완전한 구조화 프롬프트 — 권장 순서
# 1. 역할 (맨 먼저)
<role>클로드의 페르소나</role>

# 2. 맥락 (배경 정보)
<context>상황, 목적, 독자 정보</context>

# 3. 문서 · 데이터 (처리할 원자료)
<documents>
  <document index="1">참고 문서 내용</document>
</documents>

# 4. 예시 (원하는 결과물 스타일)
<examples>
  <example>원하는 출력 예시</example>
</examples>

# 5. 지시사항 (핵심 작업 — 문서 다음에 배치 시 성능 30%↑)
<instruction>해야 할 일</instruction>

# 6. 제약 조건
<constraint>하지 말 것, 범위 제한</constraint>

# 7. 출력 형식 (맨 마지막)
<format>형식, 길이, 구조</format>

P-05복사해서 바로 쓰는 프롬프트 — 여러 문서 비교 분석

Section 5 · 직업별 완성형 템플릿

직업별 완성형 XML 프롬프트 템플릿

아래 템플릿을 복사해서 기울임꼴 부분만 바꾸면 바로 사용 가능합니다.

템플릿 1: 마케터 — 캠페인 기획 프롬프트

P-06복사해서 바로 쓰는 프롬프트 — 마케터용 캠페인 기획
<role> 너는 12년 경력 B2C 디지털 마케터야. 바이럴 캠페인 기획과 SNS 콘텐츠 전략에 강해. </role> <context> 브랜드: [브랜드명] 업종: [업종] 타겟: [주 타겟 고객층] 이번 캠페인 목표: [예: 신제품 인지도 30% 향상] 예산: [예: 500만원] 기간: [예: 4주] </context> <constraint> 인플루언서 비용: 전체 예산의 30% 이내 채널: 인스타그램 + 유튜브 쇼츠만 운영 인원: 1명 </constraint> <instruction> 위 조건에 맞는 4주 캠페인 기획안을 작성해줘. 주차별 핵심 콘텐츠와 KPI를 포함해줘. </instruction> <format> 주차별 계획표 (주차 | 핵심 활동 | 기대 KPI) 그 아래 예산 배분 표 마지막으로 성공 측정 기준 3가지 </format>

템플릿 2: 개발자 — 코드 리뷰 프롬프트

P-07복사해서 바로 쓰는 프롬프트 — 개발자용 코드 리뷰
<role> 너는 10년 경력의 시니어 소프트웨어 엔지니어야. 보안 취약점과 성능 최적화에 특히 강해. </role> <context> 언어: [예: Python 3.12] 프레임워크: [예: FastAPI] 이 코드는 [어디에 사용되는 코드인지]에 사용되는 거야. 현재 발생하는 문제: [오류 메시지나 예상되는 문제] </context> <code> [여기에 리뷰받을 코드를 붙여넣으세요] </code> <instruction> 이 코드를 리뷰하고 다음을 분석해줘: 1. 버그나 오류 가능성 2. 보안 취약점 3. 성능 개선 포인트 4. 코드 가독성 개선 제안 </instruction> <constraint> 수정된 코드는 반드시 코드 블록(```)으로 제시해줘. 각 수정 사항마다 왜 변경했는지 한 줄 설명 포함. </constraint> <format> 심각도 순으로 정렬: 🔴 Critical → 🟡 Warning → 🟢 Suggestion 마지막에 수정된 전체 코드 제공 </format>

템플릿 3: 교사·강사 — 수업 자료 생성 프롬프트

P-08복사해서 바로 쓰는 프롬프트 — 교사용 수업 자료 생성
<role> 너는 경험 많은 교육 과정 설계자야. 특히 어려운 개념을 쉽게 설명하고 학생 참여를 높이는 수업을 설계하는 데 강해. </role> <context> 학교급/과목: [예: 고등학교 화학] 대상 학년: [예: 고1] 수업 주제: [예: 화학 결합의 종류] 수업 시간: [예: 50분] 특이 사항: [예: 실험 장비 없음, 개념 이해 위주] </context> <examples> <example> 이런 식의 비유를 선호해: "이온 결합은 레고 블록처럼 딱 맞는 조각끼리 결합하는 것" </example> </examples> <instruction> 이 수업을 위한 50분 수업 지도안을 만들어줘. - 도입(5분): 흥미 유발 질문 - 전개(35분): 핵심 개념 3가지 + 비유 - 마무리(10분): 확인 퀴즈 3문제 각 단계별 교사 발화 예시도 포함해줘. </instruction> <constraint> 고1 수준에 맞는 어휘 사용. 수식/공식 최소화, 직관적 이해 중심. </constraint>

템플릿 4: 작가·에디터 — 글 피드백 프롬프트

P-09복사해서 바로 쓰는 프롬프트 — 작가용 글 피드백
<role> 너는 10년 경력의 논픽션 에디터야. 글의 구조와 흐름을 개선하는 데 강하고, 필자의 고유한 목소리는 절대 지우지 않는 원칙이 있어. </role> <context> 글의 종류: [예: 브런치 에세이] 독자: [예: 30~40대 직장인] 글쓴이 의도: [예: 번아웃 경험과 회복 과정 공유] 이 글의 목표: [예: 공감과 위로, 실용적 팁 전달] </context> <text> [여기에 피드백 받을 글을 붙여넣으세요] </text> <instruction> 이 글에 대한 에디터 피드백을 주줘: 1. 가장 잘 된 부분 (구체적 문장 인용) 2. 구조/흐름 개선이 필요한 부분 3. 독자 관점에서 더 명확해야 할 부분 4. 제목 대안 3개 </instruction> <constraint> 문장을 통째로 다시 쓰지 말 것. 내 문체와 말투를 바꾸지 말 것. AI 특유의 매끄러운 문체로 수정 제안 금지. </constraint>

템플릿 5: 연구자·학생 — 논문 분석 프롬프트

P-10복사해서 바로 쓰는 프롬프트 — 연구자용 논문 분석
<role> 너는 [해당 분야] 분야의 리서치 어시스턴트야. 논문을 빠르게 읽고 핵심을 추출하는 능력이 뛰어나. </role> <context> 나는 [전공 및 연구 주제]를 연구 중인 [학년/직위]야. 이 논문을 읽는 목적: [예: 문헌 리뷰를 위한 배경 파악] </context> <document source="분석할 논문"> [여기에 논문 전문 또는 초록+본문 일부를 붙여넣으세요] </document> <instruction> 이 논문을 분석해줘: 1. 연구 질문 / 가설 2. 연구 방법론 요약 3. 핵심 발견 3가지 4. 연구의 한계점 5. 내 연구와의 관련성 </instruction> <constraint> 논문에 없는 내용은 절대 추가하지 말 것. 확실하지 않은 해석은 "(불확실)" 표시. 인용 시 페이지/섹션 정보 포함. </constraint> <format> 각 항목을 소제목과 함께 구조화. 핵심 문장은 논문에서 직접 인용(원문 + 한국어 번역). </format>
Section 6 · 고급 기법

고급 기법 — 출력 제어 & 재사용 가능한 변수 템플릿

XML 태그를 마스터하면 클로드의 응답 구조까지 제어하고, 반복 업무를 자동화할 수 있습니다.

기법 1: 출력에 XML 태그 사용 요청 — 구조화된 응답 받기

입력 프롬프트에만 XML 태그를 쓰는 것이 아니라, 클로드의 응답도 특정 태그로 감싸달라고 요청할 수 있습니다. 응답을 자동으로 처리하거나 파싱할 때 매우 유용합니다.

P-11복사해서 바로 쓰는 프롬프트 — 출력에 태그 지정 요청
<context> 아래 제품 리뷰 3개를 분석해줘. </context> <reviews> <review id="1">배송이 빠르고 품질이 좋아요. 다만 포장이 아쉬웠어요.</review> <review id="2">가격 대비 최고입니다! 재구매 의사 100%</review> <review id="3">기대보다 실망스러웠어요. CS 대응이 너무 느렸습니다.</review> </reviews> <instruction> 각 리뷰를 분석하고 아래 형식으로 결과를 XML 태그로 감싸서 응답해줘: </instruction> <output_format> <analysis id="리뷰번호"> <sentiment>긍정/부정/중립</sentiment> <key_point>핵심 내용 한 줄</key_point> <action_needed>yes/no</action_needed> </analysis> 위 형식을 각 리뷰마다 반복해줘. </output_format>

기법 2: 변수 템플릿 — 반복 업무 자동화의 기초

중괄호 {{변수명}}을 사용해서 매번 바꾸는 부분만 교체하는 재사용 템플릿을 만들 수 있습니다. 이메일, 보고서, 공지문 등 형식이 반복되는 업무에 강력합니다.

P-12복사해서 바로 쓰는 프롬프트 — 변수 템플릿으로 이메일 자동 생성
# 이 템플릿을 저장해두고, {{변수}} 부분만 바꿔서 반복 사용하세요 <role>너는 비즈니스 커뮤니케이션 전문가야.</role> <variables> 수신자: {{수신자 이름과 직함}} 발신자: {{발신자 이름과 회사}} 이메일 목적: {{이메일 목적}} 핵심 메시지: {{전달할 핵심 내용}} 원하는 행동: {{상대방에게 기대하는 액션}} 마감/일정: {{관련 날짜나 기한}} </variables> <instruction> 위 변수를 바탕으로 비즈니스 이메일을 작성해줘. 제목 포함. 정중하지만 명확하게. </instruction> <constraint> 300자 이내. 핵심 메시지는 첫 문단에. 행동 유도는 구체적으로 (날짜, 방법 포함). </constraint>
✅ 변수 템플릿 활용 예시 — 어떻게 쓰는가
위 P-12 템플릿에서 {{변수}} 부분만 바꾸면 됩니다:

미팅 일정 요청 이메일: 수신자: "김OO 팀장님", 목적: "다음 주 프로젝트 킥오프 미팅 일정 조율", 원하는 행동: "가능한 시간 2~3개 회신 요청"

기법 3: 클로드에게 XML 구조 만들어달라고 요청하기

XML 태그 구조를 직접 짜기 어려울 때, 클로드에게 먼저 구조를 제안해달라고 할 수 있습니다.

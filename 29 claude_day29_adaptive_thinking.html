📖 핵심 원리 이해

클로드가 "생각하는" 두 가지 방식

일반적인 AI는 입력을 받으면 즉시 출력을 생성합니다. 하지만 클로드 Opus 4.6과 Sonnet 4.6은 달리 작동합니다. 어려운 문제를 받으면 먼저 내부에서 사고(Thinking) 과정을 거친 후 답변을 생성합니다. 마치 복잡한 수학 문제를 풀 때 계산 과정을 먼저 적어보는 것처럼요.

적응형 사고(Adaptive Thinking)는 클로드가 스스로 판단합니다. "이 질문에 깊은 사고가 필요한가?" — 필요하면 사고하고, 단순하면 즉답합니다. 2026년 2월 이후 Opus 4.6의 기본 모드이며, 이전의 "예산 토큰 지정" 방식을 대체합니다.

L1

기본 활용 — Claude.ai에서 사고 깊이 조절하기

⏱ 예상 시간 25분  |  Claude.ai 웹/앱 사용자

코딩이나 API 없이도 Claude.ai에서 적응형 사고의 혜택을 받을 수 있습니다. 문제를 어떻게 제시하느냐에 따라 클로드가 더 깊이 생각하거나 빠르게 답합니다. 그리고 Extended Thinking 모드를 켜면 사고 과정 자체를 볼 수 있습니다.

적용 방법 | 프롬프트로 사고 깊이를 유도하는 5가지 패턴
  1. 1
    "단계별로 생각해줘" — 사고 과정 외부화 복잡한 문제 앞에 "이 문제를 단계별로 생각해줘"를 추가하면 클로드가 사고 과정을 명시적으로 거칩니다. Extended Thinking이 없어도 논리적 단계가 포함된 더 깊은 답변을 유도합니다.
  2. 2
    "가장 어려운 부분부터 분석해줘" — 핵심 집중 단순히 "분석해줘"보다 "이 문제에서 가장 까다로운 엣지 케이스나 모순을 먼저 찾아줘"처럼 어려운 부분에 집중하도록 유도하면 더 깊은 사고가 발동됩니다.
  3. 3
    다중 관점 요청 — 사고 범위 확장 "이 결정의 장·단점을 3가지씩 나열하고, 내가 놓치고 있는 관점도 찾아줘"처럼 여러 각도를 탐색하도록 요청하면 클로드가 더 넓게 생각합니다.
  4. 4
    불확실성 인정 요청 — 정직한 사고 "확실하지 않은 부분이 있으면 솔직히 알려줘"라고 하면 클로드가 더 신중하게 사고하고, 확신이 없는 부분을 명확히 표시하는 더 정직한 답변을 생성합니다.
  5. 5
    반례 검토 요청 — 사고 검증 "네 답변이 틀릴 수 있는 경우를 먼저 찾아봐줘"처럼 자신의 추론에 의문을 제기하도록 요청하면 더 강건한 사고 과정이 발동됩니다.

Extended Thinking 모드 — 사고 블록 예시

🧠 클로드의 사고 과정 (Thinking Block) — 이런 형태로 표시됩니다
[사고 과정 요약 - 접어서 볼 수 있음] 이 문제는 여러 제약 조건이 있어서 신중하게 접근해야겠다. 먼저 시간 복잡도 요구사항을 확인하고... O(n log n)이 요구되는데 정렬 기반 접근이 맞다. 그런데 중복 요소가 있을 경우를 고려해야 한다... 해시맵을 쓰면 O(n)으로 더 효율적이지만 공간 복잡도가... 두 가지 트레이드오프를 비교해보면...

👆 이 사고 블록이 클로드가 내부에서 실제로 "생각한" 과정의 요약입니다. 클릭하면 펼쳐집니다. 사고 블록이 있는 답변이 없는 답변보다 더 신뢰할 수 있는 이유입니다.

📋 복사해서 바로 쓰는 프롬프트 #1

단순 vs 심화 — 사고 유도 비교 실험
단순 버전 (얕은 사고 유도)
우리 팀이 Python을 써야 할까, Go를 써야 할까? → 이 질문은 클로드가 깊게 생각하지 않을 수 있음
심화 버전 (깊은 사고 유도)
우리 팀의 다음 백엔드 프로젝트에서 Python과 Go 중 어떤 언어를 선택해야 할지 결정해야 해. 이 문제를 아래 방식으로 접근해줘: 1. 먼저 이 결정에서 가장 중요한 요소가 무엇인지 파악해줘 2. 내가 제공한 정보에서 놓치고 있는 맥락이 있으면 먼저 질문해줘 3. Python과 Go 각각의 장·단점을 우리 상황에 맞게 분석해줘 4. 어느 쪽이 더 나은지 결론 짓기 어려운 요소는 무엇인지 솔직히 말해줘 5. 최종 권장안을 이유와 함께 제시하되, 틀릴 수 있는 시나리오도 함께 우리 팀 상황: - 팀원 5명: Python 경험자 3명, Go 경험자 0명 - 서비스 규모: 초기 스타트업, 트래픽 아직 작음 - 우선순위: 빠른 출시 > 성능 최적화 - 장기 계획: 2년 내 대용량 처리 필요할 수 있음

📋 복사해서 바로 쓰는 프롬프트 #2

사고 트리거 문장 모음
어떤 프롬프트 앞에도 붙여 쓸 수 있는 사고 유도 문구
━━━ 깊은 분석이 필요할 때 ━━━ 이 문제를 충분히 생각한 후 답해줘. 특히 내가 놓치고 있을 수 있는 중요한 관점을 먼저 파악해줘. ━━━ 여러 관점이 필요할 때 ━━━ 이 결정을 내리기 전에: 1) 지지하는 최강의 논거 2) 반대하는 최강의 논거 3) 두 관점을 종합한 균형 잡힌 결론 순서로 생각해줘. ━━━ 정확도가 중요할 때 ━━━ 답하기 전에 이 질문의 숨겨진 복잡성이 있는지 확인해줘. 확실하지 않은 부분은 반드시 [불확실]로 표시해줘. 틀릴 수 있는 경우도 솔직하게 포함해줘. ━━━ 창의적 해결책이 필요할 때 ━━━ 일반적인 접근법 2가지와 함께, 대부분의 사람이 생각하지 않는 참신한 접근법 1가지를 추가로 제시해줘. 왜 그 접근법이 간과되는지도 설명해줘. ━━━ 코드/기술 문제에서 ━━━ 해결책을 제시하기 전에: - 이 문제에서 가장 까다로운 엣지 케이스 파악 - 내 접근 방식의 잠재적 약점 식별 - 그런 다음 최적의 해결책 제안

심화 제어 — 사고 과정을 읽고 활용하기

⏱ 예상 시간 25분  |  Extended Thinking 심층 활용

Extended Thinking의 진짜 가치는 단순히 "더 좋은 답"이 아닙니다. 클로드의 사고 과정을 직접 읽을 수 있다는 것입니다. 사고 블록에서 어디서 헤맸는지, 어떤 가정을 했는지, 무엇을 고려했는지를 보고 더 나은 후속 질문을 할 수 있습니다.

적용 방법 | 사고 블록을 활용하는 고급 기법
  1. 1
    사고 블록의 "막힌 부분"에 집중하라 클로드가 사고 과정에서 "이 부분은 불명확하다", "두 가지 가능성이 있다"고 쓴 부분을 찾습니다. 그 부분이 실제로 가장 중요한 핵심입니다. 그 지점에 대한 추가 컨텍스트를 제공하면 답의 품질이 크게 향상됩니다.
  2. 2
    "사고 과정을 보여줘"라고 명시적으로 요청하라 Extended Thinking이 꺼진 상태에서도 "네가 이 결론에 도달한 추론 과정을 단계별로 설명해줘"라고 하면 사고 과정을 일반 텍스트로 외부화할 수 있습니다. 투명성이 중요한 의사결정에 활용하세요.
  3. 3
    사고 품질로 답변 신뢰도 평가하기 사고 블록이 짧고 단순하면 클로드가 이 문제를 쉽다고 판단한 것입니다. 하지만 우리가 어렵다고 생각하는 문제라면, 더 복잡한 컨텍스트를 추가해서 재시도하거나 "이 문제가 생각보다 복잡한 이유"를 먼저 설명해보세요.
  4. 4
    반복적 사고 심화 — 후속 질문 전략 첫 답변에서 사고 블록을 읽고, 그 안에서 발견한 가정이나 불확실성을 추가 질문으로 전달합니다. "네 사고 과정에서 X를 가정했는데, 실제로 Y라면 어떻게 달라지는지 다시 생각해줘"처럼요.

사고 블록이 생성되는 복잡한 문제 예시

Extended Thinking 없이도 사고 과정 보기
다음 의사결정 문제에 대해 답하기 전에 네 추론 과정을 먼저 보여줘. 형식은 아래처럼: 문제: [의사결정 문제 입력] 요청 형식: **[1단계: 문제 정의]** 이 결정의 핵심이 무엇인지, 어떤 불확실성이 있는지 **[2단계: 가정 목록]** 답변을 위해 내가 하고 있는 가정들 (명시적으로) **[3단계: 옵션 탐색]** 고려할 수 있는 선택지 3가지 이상 **[4단계: 각 옵션 평가]** 각각의 강점, 약점, 리스크 **[5단계: 결론]** 최종 권장사항과 이유. 내가 틀릴 수 있는 조건도 포함 이 형식을 유지하되, 각 단계에서 충분히 생각해줘. 빠른 답변보다 정확한 답변이 중요한 상황이야.
L3

API 설계 — effort 파라미터로 비용·품질 최적화

⏱ 예상 시간 25분  |  개발자·API 사용자

API로 클로드를 사용하는 개발자라면 effort 파라미터로 사고 깊이를 제어합니다. 모든 요청에 최대 사고를 적용하면 비용이 폭발합니다. 적절한 effort 수준을 작업 유형별로 설정하는 것이 비용 최적화의 핵심입니다.

⚠️ API 사용자를 위한 중요 변경 사항 (2026년) Opus 4.6과 Sonnet 4.6에서 budget_tokens는 Deprecated(더 이상 권장 안 함)입니다. 대신 thinking: {"type": "adaptive"}와 effort 파라미터를 사용하세요. 기존 코드는 아직 작동하지만 향후 제거될 예정입니다.
적용 방법 | API effort 파라미터 최적화 전략
  1. 1
    작업 유형별 effort 레벨 매핑 모든 API 요청에 같은 effort를 적용하지 마세요. 분류 작업이나 단순 요약은 low, 일반 코드 작성은 medium, 아키텍처 설계나 복잡한 디버깅은 high를 적용합니다.
  2. 2
    Interleaved Thinking으로 에이전트 성능 향상 Adaptive 모드는 도구 호출 사이에도 자동으로 사고를 끼워 넣습니다(Interleaved Thinking). 에이전트 워크플로우에서 더 정확한 도구 선택과 결과 해석이 가능합니다. 수동 설정 없이 자동 활성화됩니다.
  3. 3
    system 프롬프트로 사고 빈도 조절 클로드가 너무 자주 또는 너무 드물게 사고하면 system 프롬프트로 조절할 수 있습니다. "간단한 쿼리에는 바로 답하고, 복잡한 추론이 필요한 경우에만 사고 과정을 거쳐줘" 같은 지시가 효과적입니다.
🎬 실습 시나리오 03
작업 유형별 effort를 적용한 비용 최적화 파이프라인 설계
여러 종류의 작업을 처리하는 API 파이프라인에서, 작업 복잡도에 따라 effort를 다르게 적용해서 비용과 품질을 동시에 최적화하는 패턴을 설계합니다.
  • 처리할 작업 유형 목록 작성 (분류/요약/분석/코드 등)
  • 각 작업 유형에 적합한 effort 레벨 결정
  • 아래 코드 패턴 참고해서 실제 파이프라인 구현
  • low vs high effort 비용 차이를 실제로 테스트해서 측정

📋 복사해서 바로 쓰는 프롬프트 #6

API — Adaptive Thinking 코드 패턴
Python SDK로 effort 파라미터 활용하기
# Adaptive Thinking API 사용 패턴 (2026년 권장 방식) import anthropic client = anthropic.Anthropic() # ── 작업 복잡도별 effort 레벨 매핑 ────────────────── EFFORT_MAP = { "classification": "low", # 분류, 태깅 "summarization": "low", # 단순 요약 "code_generation": "medium", # 일반 코드 작성 "code_review": "high", # 코드 리뷰, 디버깅 "architecture": "high", # 설계 의사결정 "math_proof": "high", # 수학/논리 증명 } def call_claude(task_type: str, prompt: str): effort = EFFORT_MAP.get(task_type, "medium") response = client.messages.create( model="claude-opus-4-6", max_tokens=16000, thinking={ "type": "adaptive", # ← 2026년 권장 방식 }, output_config={ "effort": effort # ← low / medium / high }, messages=[{"role": "user", "content": prompt}] ) return response # ── Claude Code에서 사용할 경우 (/effort 명령) ────── # > /effort low ← 빠른 코딩 작업 # > /effort medium ← 일반 개발 (기본값) # > /effort high ← 코드 리뷰, 디버깅 # > /effort max ← 아키텍처 설계 (Opus 4.6 전용)

System 프롬프트 — 사고 빈도 조절
API system 프롬프트로 적응형 사고 동작 제어
# system 파라미터에 추가할 사고 빈도 조절 가이드 # ── 사고를 줄이고 싶을 때 ──────────────────────── SYSTEM_LESS_THINKING = """ 간단하고 직접적인 질문에는 즉시 답변하세요. 사실 확인, 간단한 번역, 기본 정보 제공 등은 사고 과정 없이 바로 응답하세요. 다음 유형의 질문에만 깊은 사고를 적용하세요: - 여러 상충 요소가 있는 의사결정 - 복잡한 코드 디버깅 또는 아키텍처 설계 - 수학적/논리적 증명 필요한 경우 """ # ── 사고를 더 많이 유도하고 싶을 때 ───────────── SYSTEM_MORE_THINKING = """ 답변 전에 항상 다음을 점검하세요: 1. 이 질문의 숨겨진 복잡성이 있는가? 2. 내가 놓치고 있는 중요한 가정이 있는가? 3. 더 나은 접근법이 있는가? 불확실한 부분은 반드시 [불확실]로 표시하고, 틀릴 수 있는 조건을 명시하세요. """ # ── 에이전트 워크플로우용 ──────────────────────── SYSTEM_AGENT = """ 도구를 사용한 후 결과를 신중하게 평가하세요. 예상치 못한 결과가 나오면 즉시 다음 도구를 호출하지 말고, 현재 상황을 재평가하고 최선의 다음 행동을 결정하세요. """

적응형 사고란?

클로드가 스스로 사고 필요성과 깊이를 판단하는 모드. Opus 4.6 기본값. 이전 budget_tokens 방식을 대체.

4단계 사고 스펙트럼

Low(즉답) → Medium(기본 사고) → High(깊은 사고, 기본값) → Max(최대 사고, Opus 전용)

Claude.ai 활용

Extended Thinking 토글 ON으로 사고 블록 확인 가능. 사고 블록의 "막힌 부분"에 추가 컨텍스트 제공이 핵심.

API 최적화

thinking: {"type": "adaptive"} + effort 파라미터. 작업 유형별 매핑으로 비용 절감. system 프롬프트로 사고 빈도 조절.

사고 깊이 유도 패턴

"단계별로", "막힌 부분", "반례 검토", "불확실성 인정" 요청이 효과적. 복잡한 프롬프트 = 더 깊은 사고.

Projects란? — 일반 대화와 무엇이 다른가

클로드를 매번 "새로운 직원"처럼 대하는 것에서 벗어나, 나의 업무와 전문 지식을 완전히 파악한 "숙련된 파트너"로 만드는 기능입니다.

일반 대화 vs Projects — 핵심 차이

❌ 일반 대화 (대화별 리셋)
매번 새 대화 → 클로드가 나를 모름
"나는 마케터야, 우리 제품은..." 반복 입력
파일 업로드해도 다음 대화에서 사라짐
지시사항을 매번 처음부터 설명
→ 시간 낭비 + 일관성 없는 답변
✅ Projects (영구 컨텍스트)
프로젝트 안에서는 클로드가 항상 맥락 파악
문서·데이터가 프로젝트에 영구 저장됨
지시사항(Instructions)이 모든 대화에 자동 적용
이전 대화 내용을 기반으로 쌓이는 지식
→ 진짜 전문가 파트너처럼 작동

Projects의 3요소 해부

📁 클로드 프로젝트의 구조
📋
Instructions (지시사항)
클로드에게 역할·규칙·행동 방침을 영구적으로 설정. 모든 대화에 자동 적용되는 시스템 프롬프트.
📚
Knowledge (지식 기반)
PDF, 문서, 데이터, 코드 등 업로드한 파일들의 집합. 클로드가 언제든 참조하는 전문 지식 저장소.
💬
Conversations (대화)
프로젝트 안에서 나누는 모든 대화. 컨텍스트가 이어지며 대화가 쌓일수록 클로드가 더 정확해짐.

어떤 프로젝트가 가능한가?

💼
업무 전문가
내 직무 매뉴얼·프로세스 문서를 업로드 → 업무 질문에 항상 우리 회사 기준으로 답변
📖
연구 어시스턴트
논문·보고서 집합 → "이 연구들을 기반으로 분석해줘"가 가능한 학술 파트너
🛒
제품 전문가
제품 스펙·FAQ·정책 업로드 → 고객 문의에 정확한 제품 정보로 답변 생성
✍️
글쓰기 파트너
내 기존 글·브랜드 가이드라인 → 항상 내 목소리와 스타일을 유지하는 글쓰기 보조
💻
코딩 어시스턴트
코드베이스·아키텍처 문서 → 프로젝트 전체를 이해한 상태에서 코딩 지원
🎓
학습 튜터
교재·강의 노트 → 내가 공부하는 내용을 완전히 파악한 개인 맞춤 튜터
💡 무료 플랜에서도 Projects 사용 가능
Projects는 무료 플랜에서도 최대 5개까지 만들 수 있습니다. Pro 이상 플랜에서는 더 많은 프로젝트와 대용량 파일 업로드가 가능합니다. 가장 먼저 만들 프로젝트 하나를 지금 바로 시작해보세요.
✅ 적용방법 — Projects 체험하기 전 준비
프로젝트를 만들기 전에, 아래 프롬프트로 내 상황에 맞는 프로젝트 전략을 먼저 설계합니다.

Section 1 · 프로젝트 생성

📁 프로젝트 만들기 — 단계별 설정 가이드

처음 프로젝트를 만드는 과정을 화면 단위로 따라합니다. 5분이면 첫 프로젝트가 완성됩니다.

1-1. 프로젝트 생성 단계

사이드바에서 "Projects" 또는 "+" 버튼 찾기
claude.ai 왼쪽 사이드바에서 Projects 섹션을 찾습니다. "New Project" 또는 "+" 버튼을 클릭합니다. 모바일 앱에서는 ≡ 메뉴 → Projects 순으로 접근합니다.
프로젝트 이름 입력
명확하고 구체적인 이름을 사용하세요. "마케팅 전략 2026", "고객 응대 매뉴얼", "논문 리서치"처럼 나중에 봐도 무슨 프로젝트인지 알 수 있게.
Instructions 작성 (핵심!)
이 프로젝트에서 클로드가 어떻게 행동할지 지정합니다. 역할, 어조, 금지 사항, 출력 형식 등을 포함합니다. (자세한 작성법은 Section 3에서 배웁니다)
문서 업로드 (Knowledge 구축)
프로젝트와 관련된 PDF, Word, 텍스트 파일을 업로드합니다. 파일은 영구 저장되어 이후 모든 대화에서 클로드가 참조합니다. (업로드 전략은 Section 2에서)
첫 번째 대화 시작
프로젝트 안에서 "New Conversation"을 클릭해서 첫 대화를 시작합니다. Instructions와 업로드한 문서가 이미 적용된 상태로 대화가 시작됩니다.

Section 2 · 지식 기반(KB) 구축

📚 지식 기반 구축 — 문서 업로드 전략

어떤 문서를 어떻게 업로드하느냐에 따라 클로드가 얼마나 정확하게 답변하는지가 결정됩니다.

2-1. 업로드 가능한 파일 형식

PDF
PDF 문서
보고서, 논문, 매뉴얼, 계약서 — 가장 범용적
DOCX
Word 문서
내부 정책, 기획서, 회의록, 절차서
TXT
텍스트 파일
로그, 메모, 코드 스니펫, 설정 파일
MD
Markdown
README, 기술 문서, 위키 내용
CSV
데이터 파일
통계 데이터, 고객 목록, 제품 카탈로그
기타
이미지·코드
PNG/JPG 이미지, .py .js .html 코드 파일
XLSX
Excel 파일
스프레드시트, 예산표, 데이터 분석
기타
PPTX 등
발표 자료, 슬라이드 내용 분석

2-2. 효과적인 문서 업로드 전략

전략 1: 핵심 문서 우선 — "80/20 원칙"
모든 문서를 다 올리는 것보다, 클로드가 가장 자주 참조할 핵심 문서 2~5개를 엄선하세요. 너무 많은 문서는 오히려 클로드의 답변 정확도를 낮출 수 있습니다.
전략 2: 문서 품질 높이기 — 업로드 전 정제
스캔된 PDF(이미지 기반)보다 텍스트 기반 PDF가 훨씬 잘 인식됩니다. 업로드 전에 텍스트 복사가 되는지 확인하세요. OCR이 적용된 스캔 문서는 품질이 떨어질 수 있습니다.
전략 3: 문서 구조 명확히 — 제목·목차·섹션
클로드는 문서의 구조를 파악해서 관련 부분을 찾아냅니다. 헤더, 섹션 제목, 목차가 명확한 문서일수록 참조 정확도가 높아집니다.
전략 4: 문서 설명 제공 — Instructions에 명시
Instructions에 "doc1.pdf는 2026년 제품 로드맵입니다. doc2.pdf는 경쟁사 분석 보고서입니다"처럼 각 문서가 무엇인지 설명하면 클로드가 상황에 맞는 문서를 먼저 참조합니다.
전략 5: 정기 업데이트 — 지식 기반 최신화
업로드한 문서는 자동으로 업데이트되지 않습니다. 새 버전 문서가 나오면 기존 파일을 삭제하고 새 파일을 업로드하세요. 날짜를 파일명에 포함하면 관리가 편합니다.
🎭 실습 시나리오 2
상황: 고객 지원팀의 Projects를 만들려고 합니다. 제품 FAQ, 정책 문서, 응대 가이드라인을 업로드하고 클로드가 이를 기반으로 고객 응대 초안을 작성해줬으면 합니다.
목표: 문서 업로드 후 클로드가 문서를 잘 이해했는지 확인하기
P-02복사해서 바로 쓰는 프롬프트 — 문서 업로드 후 이해도 확인 테스트
# 업로드한 문서들을 클로드가 잘 파악했는지 테스트 방금 업로드한 문서들을 검토하고 아래 질문들에 답해줘. ① 각 문서의 핵심 내용 3줄 요약 (문서별로) ② 문서들 사이에 내용이 겹치거나 충돌하는 부분이 있으면 알려줘 ③ 문서에서 불명확하거나 더 정보가 필요한 부분은? ④ 이 문서들을 바탕으로 내가 자주 요청할 것 같은 질문 3가지와 예상 답변을 짧게 그리고 테스트 질문: "[업로드한 문서와 관련된 실제 질문 하나]" 이 질문에 답할 때, 어떤 문서의 몇 번째 섹션을 참조했는지도 알려줘.
P-03복사해서 바로 쓰는 프롬프트 — 문서에서 특정 정보 추출하기
# 업로드된 문서에서 필요한 정보만 골라내기 업로드된 문서들에서 아래 정보를 찾아서 정리해줘. [찾을 정보] ① [예: 환불 정책의 세부 조건과 기한] ② [예: 제품 보증 기간 및 적용 범위] ③ [예: 고객 응대 시 사용해야 할 공식 표현들] [정리 형식] 각 정보마다: - 핵심 내용 (2~3문장) - 출처 (문서명 + 해당 섹션) - 주의사항 또는 예외 조건 [추가 요청] 위 정보들을 바탕으로 고객이 "환불 요청"을 해왔을 때 우리 정책에 맞는 응대 시나리오 1개를 예시로 작성해줘.
Section 3 · Instructions 작성법

📋 Instructions — 클로드를 내 전문가 파트너로 만들기

Instructions는 프로젝트의 두뇌입니다. 잘 작성된 Instructions 하나가 수십 번의 프롬프트를 대체합니다.

좋은 Instructions의 6가지 구성 요소

요소내용예시
① 역할 정의클로드의 페르소나와 전문성"너는 [회사명]의 B2B 마케팅 전략 어시스턴트야"
② 목적·용도이 프로젝트의 목적"이 프로젝트의 목적은 주간 마케팅 보고서 작성 지원"
③ 지식 베이스 안내업로드한 문서 설명"[문서명]은 2026 제품 로드맵, [문서명]은 경쟁사 분석"
④ 응답 규칙언어·형식·길이·어조"항상 한국어로, 비즈니스 어조, 500자 이내"
⑤ 금지 사항하면 안 되는 것"문서에 없는 내용 추측 금지, 확실하지 않으면 명시"
